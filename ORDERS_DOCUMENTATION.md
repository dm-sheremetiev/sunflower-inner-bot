# Документация по работе с заказами

## Обзор

Система обрабатывает вебхуки от KeyCRM и автоматизирует уведомления в Telegram, а также отправку фото клиентам при изменении статусов заказов.

## Архитектура

### API Клиенты

Система использует два API клиента для работы с KeyCRM:

#### 1. `keycrmApiClient`
- **Назначение**: Основной API клиент для работы с заказами
- **Аутентификация**: Bearer токен через `KEYCRM_API_KEY`
- **Base URL**: `KEYCRM_API_URL`
- **Использование**: Получение информации о заказах, изменение статусов, работа с тегами

#### 2. `keycrmAdminApiClient`
- **Назначение**: Админ API для расширенных операций
- **Аутентификация**: Bearer токен через логин/пароль (автоматическое обновление)
- **Base URL**: `KEYCRM_ADMIN_API_URL`
- **Использование**: Получение полной информации о заказах с attachments, отправка сообщений в чаты клиентов
- **Особенности**: 
  - Автоматическое обновление токена при 401 ошибке
  - Логирование всех запросов и ответов

## Основные функции работы с заказами

### Получение информации о заказе

```typescript
getOrderInfo(orderId: string | number, reply: FastifyReply)
```

Получает полную информацию о заказе через основной API:
- Включает: assigned, custom_fields, shipping.deliveryService, buyer, manager, products, tags
- URL: `order/${+orderId}` (унарный плюс для нормализации ID)

### Изменение статуса заказа

```typescript
changeOrderStatus(orderId: number | string, status_id: string)
```

Изменяет статус заказа через PUT запрос:
- URL: `order/${+orderId}`
- Body: `{ status_id: +status_id }`

### Работа с тегами

```typescript
addTagToOrder(ctx, orderId, bot, extraArgument)
```

Добавляет тег "БУКЕТ НА СКЛАДІ" к заказу:
- Проверяет наличие тега у заказа
- Если тег отсутствует, получает список тегов и добавляет нужный
- Отправляет уведомление в Telegram группу склада

## Webhook Endpoints

Все вебхуки принимают POST запросы с телом в формате `ChangeOrderEvent`:

```typescript
{
  event: "order.change_order_status",
  context: {
    id: number,           // ID заказа
    status_id: number,     // Новый статус
    client_id: number,     // ID клиента
    manager_id: number,    // ID менеджера
    // ... другие поля
  }
}
```

### Список endpoints

#### 1. `POST /webhook/`
**Функция**: `getEvent` → `sendTelegramMessageAboutOrder`
- **Назначение**: Отправка уведомления назначенным сотрудникам о новом назначении заказа
- **Логика**:
  - Получает информацию о заказе
  - Форматирует сообщение с информацией о заказе и назначенных сотрудниках
  - Отправляет сообщение каждому назначенному сотруднику в Telegram
  - Исключает курьеров из списка получателей

#### 2. `POST /webhook/new`
**Функция**: `sendNewOrderMessage` → `sendMessageAboutNewOrder`
- **Назначение**: Уведомление о новом заказе
- **Логика**:
  - Проверяет наличие назначенных сотрудников
  - Если назначенных нет - отправляет сообщение менеджеру
  - Отправляет уведомление всем назначенным сотрудникам

#### 3. `POST /webhook/prepared-without-package`
**Функция**: `sendWithoutPackageMessage` → `sendWithoutPackageMessageToManager`
- **Назначение**: Уведомление о том, что заказ собран без упаковки
- **Логика**:
  - Отправляет сообщение менеджеру или в канал уведомлений
  - Указывает ответственных сотрудников (исключая курьеров)
  - Напоминает проверить наличие фото

#### 4. `POST /webhook/pack-me`
**Функция**: `sendPackMeMessage` → `sendMessageAboutPackage`
- **Назначение**: Уведомление о готовности заказа к упаковке
- **Логика**:
  - Отправляет сообщение всем ответственным (исключая курьеров)
  - Включает информацию о листовке и комментариях к товару
  - Если нет ответственных - уведомляет менеджера

#### 5. `POST /webhook/packed`
**Функция**: `sendPackedMessage` → `sendPackedMessageNotification`
- **Назначение**: Уведомление о том, что заказ упакован
- **Логика**:
  - Отправляет сообщение менеджеру или в канал уведомлений
  - Указывает ответственных сотрудников
  - Напоминает проверить наличие фото

#### 6. `POST /webhook/waiting-for-delivery`
**Функция**: `sendWaitingForDeliveryMessage` → `sendMessageAboutWaiting`
- **Назначение**: Детальное уведомление менеджеру о заказе, ожидающем отправки
- **Логика**:
  - Формирует подробное сообщение с:
    - Названием товара и артикулом
    - Временем для флористов и курьеров (из custom_fields)
    - Адресом доставки
    - Комментариями клиента
    - Листовкой
    - Номерами телефонов
  - Отправляет менеджеру

#### 7. `POST /webhook/file/bare-comp-message`
**Функция**: `sendBareCompositionImageHandler` → `sendImageToCustomerChat(orderId, 0)`
- **Назначение**: Отправка фото збірки без пакування клиенту
- **Логика**: См. раздел "Отправка фото клиентам"

#### 8. `POST /webhook/file/packed-comp-message`
**Функция**: `sendPackedCompositionImageHandler` → `sendImageToCustomerChat(orderId, 1)`
- **Назначение**: Отправка фото упакованного заказа клиенту
- **Логика**: См. раздел "Отправка фото клиентам"

#### 9. `POST /webhook/validate-status-change`
**Функция**: `validateStatusChangeHandler` → `validateOrderStatusChange`
- **Назначение**: Валідація зміни статусу (дата відправки, тег філії). При невідповідності — відкат на початковий статус та повідомлення в чат менеджерів.

#### 10. `POST /webhook/validate-address-revert`
**Функция**: `validateAddressAndRevertHandler` → `validateOrderAddressAndRevert`
- **Назначение**: Перевірка адреси доставки та координат. Якщо умови не виконані — замовлення відкатується на **статус id = 7** і в чат менеджерів відправляється повідомлення.
- **Виняток**: для замовлень з тегом **«Самовивіз»** або **«Уточнити адресу»** перевірку не виконують (адреса/координати не перевіряються).
- **Умови відкату**:
  1. Вказана адреса доставки (місто/регіон/вулиця/точка видачі), але **не вказані координати** (поле «індекс» у доставці).
  2. Координати вказані **не у форматі "число, число"** (наприклад `50.45, 30.52`). Інші формати не приймаються.
  3. **Адреси немає взагалі** і у поточного статусу замовлення **group_id = 4** (тобто замовлення вже в «доставковому» статусі без адреси).
- **Тіло запиту**: таке ж, як у всіх вебхуків — подія `order.change_order_status` з `context.id` = ID замовлення.

### Як підписати автоматизацію «Валідація адреси / відкат» в KeyCRM

Щоб ця перевірка спрацьовувала автоматично при зміні статусу замовлення:

1. **Відкрийте налаштування автоматизацій** у KeyCRM (розділ інтеграцій або сценарії / webhooks).
2. **Створіть нову автоматизацію** (webhook на подію зміни статусу замовлення).
3. **Подія**: `order.change_order_status` (або еквівалент «Зміна статусу замовлення»).
4. **Дія**: виклик зовнішнього URL методом **POST**:
   - URL: `https://<ваш-домен>/webhook/validate-address`
   - Тіло запиту: передавати **повне тіло події**, яке KeyCRM відправляє при зміні статусу (зазвичай це вже є в налаштуванні «Webhook URL» / «Payload»).
5. **Назва автоматизації** (щоб було зрозуміло в списку): наприклад *«Валідація адреси та координат — відкат на статус 7»* або *«Перевірка адреси доставки / індекс (координати)»*.
6. Збережіть і включіть автоматизацію.

Після цього при кожній зміні статусу замовлення KeyCRM буде викликати ваш сервіс; якщо адреса або координати не відповідають правилам — замовлення автоматично повернеться на статус з id = 7 і менеджери отримають повідомлення в Telegram.

## Отправка фото клиентам

### Функция `sendImageToCustomerChat`

**Параметры**:
- `reply: FastifyReply` - объект ответа Fastify
- `orderId: number` - ID заказа
- `attachmentIndex: number = 0` - индекс фото в массиве attachments (0 - збірка, 1 - пакування)

### Процесс работы

1. **Нормализация orderId**
   ```typescript
   const normalizedOrderId = Number(orderId);
   if (isNaN(normalizedOrderId)) {
     throw new Error("Невірний номер замовлення");
   }
   ```
   - Предотвращает проблемы с неправильным формированием URL (например, `/v1/order/164801` вместо `/v1/order/16480`)

2. **Получение информации о заказе**
   - Использует `keycrmAdminApiClient.get<AdminOrder>('/orders/${normalizedOrderId}')`
   - Получает полную информацию включая attachments

3. **Проверки**
   - Проверка существования заказа
   - Проверка тега "блогер" (для блогеров фото не отправляется)
   - Проверка наличия attachments
   - Проверка наличия файла по индексу
   - Проверка наличия URL или thumbnail у файла

4. **Получение чата клиента**
   - Получает все conversations клиента: `/conversations/by-client/${clientId}`
   - Выбирает последний активный чат (по `updated_at`)

5. **Обработка файла**
   - Если файл больше 5MB - скачивает и перезагружает в KeyCRM storage
   - Использует оригинальный URL или thumbnail если перезагрузка не удалась

6. **Отправка сообщений**
   - Сначала отправляет текстовое сообщение с описанием
   - Ждет 2 секунды
   - Отправляет фото как отдельное сообщение

7. **Обработка ошибок**
   - Все ошибки логируются
   - Отправляется уведомление в Telegram канал (`PHOTO_APPROVAL_CHAT_ID`)
   - Сообщения об ошибках содержат:
     - Номер заказа
     - Информацию о доступных файлах
     - Информацию об ответственных сотрудниках
     - Название филиала (если есть тег)

### Важные моменты

- **Нормализация orderId**: Критически важно для предотвращения ошибок в URL
- **Проверка attachments**: Система проверяет наличие файлов перед отправкой
- **Обработка больших файлов**: Автоматическая перезагрузка файлов >5MB
- **Fallback на thumbnail**: Если основной URL недоступен, используется thumbnail
- **Раздельная отправка**: Текст и фото отправляются отдельными сообщениями

## Обработка ошибок

### Safety Wrapper

Все вебхуки обернуты в `safetyWrapper`, который:
- Валидирует входящий запрос
- Извлекает `orderId` из `body.context.id`
- Проверяет тип события (`order.change_order_status`)
- Обрабатывает ошибки единообразно
- Предотвращает двойную отправку ответа

### Логирование

- Все запросы к Admin API логируются через interceptors
- Логируются успешные запросы (✅) и ошибки (❌)
- Включает URL, метод, статус ответа, тело запроса/ответа

## Работа с пользователями

Система использует файл `users.json` для маппинга:
- `chatId` (Telegram) → `username` (KeyCRM)

При отправке сообщений:
1. Получает список назначенных сотрудников из заказа
2. Ищет их `username` в `users.json`
3. Отправляет сообщение по найденному `chatId`
4. Если пользователь не найден - логирует ошибку

## Типы данных

### Order (основной API)
- Базовая информация о заказе
- Включает: buyer, manager, products, shipping, assigned, tags, custom_fields

### AdminOrder (админ API)
- Расширенная информация о заказе
- Включает: attachments, полная информация о файлах
- Используется для отправки фото клиентам

### ChangeOrderEvent
- Структура вебхука от KeyCRM
- Содержит: event type, context с полной информацией о заказе

## Переменные окружения

- `KEYCRM_API_KEY` - токен для основного API
- `KEYCRM_API_URL` - URL основного API
- `KEYCRM_ADMIN_API_URL` - URL админ API
- `KEYCRM_ADMIN_USERNAME` - логин для админ API
- `KEYCRM_ADMIN_PASSWORD` - пароль для админ API
- `PHOTO_APPROVAL_CHAT_ID` - ID Telegram канала для уведомлений об ошибках отправки фото
- `BRANCHES` - список филиалов через запятую (для извлечения названий филиалов из тегов)

## Важные исправления

### Проблема с добавлением "1" в конце номера заказа
**Проблема**: Иногда в URL добавлялась лишняя цифра (например, `/v1/order/164801` вместо `/v1/order/16480`)

**Решение**: Добавлена нормализация `orderId` через `Number(orderId)` с проверкой на `NaN` перед использованием в URL.

### Проблема с отправкой фото
**Проблемы**:
- Недостаточная проверка наличия файлов
- Отсутствие информации о количестве доступных файлов при ошибках
- Недостаточная обработка ошибок

**Решения**:
- Добавлена проверка наличия URL или thumbnail перед использованием
- Улучшены сообщения об ошибках с указанием индекса и количества файлов
- Раздельная обработка ошибок при отправке текста и фото
- Улучшена обработка больших файлов с fallback на оригинальный URL
