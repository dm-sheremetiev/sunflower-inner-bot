# Зв’язок позицій Poster з CRM (сайт + KeyCRM)

Документ описує, **як ідентифікатори з Poster** відображаються на **товари на сайті** (PostgreSQL / Prisma) і на **KeyCRM**, і як вони використовуються в **рекалькуляції цін** та **синхронізації Poster → KeyCRM**.

---

## 1. Три системи та їх роль

| Система | Роль |
|--------|------|
| **Poster** | Джерело меню та цін (`menu.getProducts`): товари, модифікатори, техкарти (`group_modifications`). |
| **Сайт (наша БД)** | Каталог для вітрини: `Product`, `ProductVariant`, склад варіанта — `ProductVariantComponent` з полями Poster. |
| **KeyCRM** | Облік номенклатури для продажів/закупівель; зв’язок з Poster через **кастомні поля** товару. |

Категорії для KeyCRM підтягуються з Poster за **`category_name`** і зіставляються з категоріями KeyCRM за **нормалізованою назвою** (trim, lower case).

---

## 2. Ідентифікатори Poster у нашій БД

У моделі **`ProductVariantComponent`** (і аналогічно **`ShowcaseItemComponent`**) зберігаються:

| Поле | Що це в Poster |
|------|----------------|
| **`external_id`** | Зазвичай **`product_id`** батьківського товару в Poster (для модифікаторів теж дублюється `product_id` товару, з якого знято модифікацію). |
| **`modificator_id`** | ID **звичайної модифікації** (`modifications[]` у відповіді API). |
| **`dish_modification_id`** | ID позиції з **техкарти** (`group_modifications[].modifications[]`, поле `dish_modification_id`). |

Один варіант товару на сайті — це **композиція** кількох компонентів; ціна на вітрині в рекалькуляції збирається як **сума цін компонентів** з Poster (з урахуванням `quantity`).

Поле **`ScheduledPriceChange`** у схемі також тримає трійку `product_id` / `modificator_id` / `dish_modification_id` для запланованих змін цін у Poster.

---

## 3. Як Poster віддає дані (логіка `menu.getProducts`)

У коді розрізняють три випадки для **одного** об’єкта `PosterProduct`:

### 3.1. Товар **без** `modifications` і **без** `group_modifications`

- Ціна з **`spots`**: береться **максимум** серед spot з `visible === "1"`, `price` ділиться на 100 (копійки → гривні).
- У мапу цін для рекалькуляції потрапляє ключ **`product_id`** (як рядок).
- Стара ціна «якоря» для акції може парситися з **`barcode`** товару (у гривнях), якщо задана.

### 3.2. Товар з **`modifications`** (звичайні модифікатори)

- Для кожного модифікатора: ціна з **`mod.spots`** (аналогічно, max по видимих spot / 100).
- У мапу цін: ключ **`modificator_id`** (тип у API — рядок).
- Стару ціну для акції беруть з **`modificator_barcode`** (або fallback).
- Якщо всі видимі spot дають **нульову** ціну — позиція вважається **прихованою** в Poster; вона реєструється окремо для сповіщень (не потрапляє в `posterPrices` як валідна ціна).

### 3.3. Товар з **`group_modifications`** (техкарти)

- Для кожної модифікації в групі: **`price` уже в гривнях** (не ділиться на 100).
- У мапу цін: ключ **`String(dish_modification_id)`**.
- Старої ціни з barcode для цього шляху в рекалькуляції **немає** (`oldPrice = null`).

### 3.4. Повністю прихований товар

- Якщо `hidden === "1"`, товар не використовується як джерело цін у рекалькуляції; ключі реєструються для діагностики **прихованих компонентів**, що можуть збігатися з компонентами на сайті.

---

## 4. Рекалькуляція цін на сайті (`recalculatePrices`)

Після запиту до Poster будується **`Map<string, { currentPrice, oldPrice }>`**, де ключ — **один** з:

- `product_id`
- `modificator_id`
- `dish_modification_id` (як рядок)

Для кожного **`ProductVariantComponent`** варіанта шукається перша співпадіння (у такому порядку):

1. `posterPrices.get(component.external_id)`
2. якщо є `modificator_id` → `posterPrices.get(component.modificator_id)`
3. якщо є `dish_modification_id` → `posterPrices.get(String(component.dish_modification_id))`

Якщо запису **немає** (компонент зник або прихований у Poster):

- варіант може бути **деактивований** (`is_active: false`);
- назва компонента може потрапити в **Telegram**, якщо це «прихований у Poster» компонент, який збігається з ключами з мапи прихованих.

Підсумкова ціна варіанта = округлена сума по компонентах; логіка **price / discount_price** використовує порівняння «старої» суми (barcode) з «поточною» (spots), щоб виставити акційну ціну там, де в Poster задано стару вищу ціну.

**Висновок:** зв’язок сайту з Poster у рекалькуляції — **не по slug товару**, а **строго по трійці полів компонента**, узгодженій з тим, як ключі заповнюються з `menu.getProducts`.

---

## 5. Синхронізація Poster → KeyCRM (`syncPosterToKeyCRM`)

KeyCRM не використовує наші внутрішні `cuid` з сайту. Зв’язок тільки через **кастомні поля** товару в KeyCRM:

| UUID поля | Призначення |
|-----------|-------------|
| **`CT_1008`** | Основний **зовнішній ID Poster** для зіставлення запису в KeyCRM з об’єктом у Poster. |
| **`CT_1014`** | **`ingredient_id`** (один або кілька через кому) з Poster для складу / техкарт. |

### 5.1. Звичайний товар Poster (без модифікацій у гілці обробки)

- **`CT_1008`** = **`product_id`**.
- **`CT_1014`** = зібрані `ingredient_id`: з поля товару та/або з масиву `ingredients[]`, відфільтровані «порожні» та `"0"`.

### 5.2. Кожен модифікатор з `modifications`

- Окремий товар у KeyCRM (або оновлення існуючого).
- **`CT_1008`** = **`modificator_id`**.
- **`CT_1014`** = `ingredient_id` модифікатора.

### 5.3. Кожна позиція з `group_modifications`

- Окремий товар у KeyCRM.
- **`CT_1008`** = **`String(dish_modification_id)`** (це **не** `product_id` батьківського товару).
- **`CT_1014`** = `ingredient_id` модифікації техкарти, якщо є.

Пошук існуючого товару в KeyCRM для оновлення йде по **`CT_1008`** (мапа `keycrmProductsMap`). Товари з `hidden` у Poster **пропускаються** (не створюються / не оновлюються в цій гілці).

---

### 5.4. Окремо: як саме зіставляється товар у KeyCRM з об’єктом у Poster

Цей блок розширює п. 5: **який запис у KeyCRM з чим у Poster вважається «тим самим»** і **звідки копіюються поля**.

#### Ключ зіставлення (єдине правило пошуку пари)

- У коді будується мапа **`keycrmProductsMap`**: ключ — **рядкове значення кастомного поля `CT_1008`** у товарі KeyCRM, значення — відповідний товар (і пов’язані дані) з KeyCRM.
- Під час обходу меню Poster для **кожної** синхронізованої позиції обчислюється **той самий** ключ — він і записується в `CT_1008` при створенні та оновленні.
- **Жодних інших ключів** (SKU сайту, `slug`, внутрішній id Prisma) у цьому кроні для зіставлення **не використовується** — тільки `CT_1008` ↔ один з ID Poster.

#### Який ID Poster стає значенням `CT_1008` (що з чим «склеюється»)

| Що прийшло з Poster у `menu.getProducts` | Значення **`CT_1008`** у товарі KeyCRM | Один рядок KeyCRM = … |
|----------------------------------------|----------------------------------------|------------------------|
| Товар обробляється як **простий** (у гілці без прив’язки до модифікаторів у цьому циклі) | **`product_id`** | Один номенклатурний товар = весь цей продукт меню |
| Елемент **`modifications[]`** | **`modificator_id`** | Один товар KeyCRM = один модифікатор |
| Елемент **`group_modifications[].modifications[]`** | **`String(dish_modification_id)`** | Один товар KeyCRM = одна позиція техкарти (**не** `product_id` батька) |

Якщо в KeyCRM у двох товарів **однаковий** `CT_1008`, логіка мапи передбачає **один** запис на ключ (перший виграє при побудові мапи; офери з тим самим полем теж можуть підсилювати прив’язку — див. `sync-poster-keycrm.ts`).

#### Звідки беруться поля товару KeyCRM (відповідність полів Poster)

| Поле / сутність у KeyCRM | Звідки в Poster (логіка синку) |
|--------------------------|--------------------------------|
| **`name`** | Простий товар: **`product_name`** (нормалізація пробілів). Модифікатор: **`modificator_name`**. Техкарта: рядок **`{product_name} - {group.name} - {mod.name}`** |
| **`price`** (ціна продажу) | Простий: з **`price`** товару (копійки → ділення на 100). Модифікатор: з **`spots`** модифікатора (перший spot / 100 у `getPriceFromModificator`). Техкарта: **`mod.price`** уже в гривнях |
| **`purchased_price`** (собівартість) | Простий: **`cost`** товару / 100. Модифікатор: **`modificator_selfprice`** / 100. Техкарта: у коді зараз фіксована заглушка **`1`** |
| **`category_id`** | Категорія KeyCRM, знайдена або створена за **`category_name`** з Poster (зіставлення по нормалізованій назві) |
| **`pictures`** | Фото товару Poster (**`photo` / `photo_origin`**) або фото модифікації техкарти (**`photo_large` / `photo_orig`**); відносні шляхи доповнюються базою **`https://sunflower.joinposter.com`** |
| **Кастомне поле `CT_1014`** | **`ingredient_id`** модифікатора / техкарти; для простого товару — **`ingredient_id`** поля товару **або** перелік **`ingredients[].ingredient_id`** через кому (без порожніх і `"0"`) |
| **`sku` / `barcode`** (де задані в коді) | Для модифікатора: **`modificator_product_code`**, **`modificator_barcode`** |

#### Алгоритм у два речення

1. **Спочатку** з KeyCRM зчитуються товари й будується словник «**значення CT_1008 → товар KeyCRM**».  
2. **Потім** для кожної відповідної позиції з Poster обчислюється ключ (`product_id` / `modificator_id` / `dish_modification_id`), і якщо він **є** в словнику — оновлюється існуючий товар KeyCRM; якщо **немає** — товар потрапляє в чергу на **створення** з тим самим `CT_1008`.

#### Що не входить у це зіставлення

- **Каталог сайту** (`Product`, `ProductVariant`, компоненти в Prisma) у кроні **`syncPosterToKeyCRM` не читається** і до KeyCRM **не прив’язується**. Зв’язок сайт ↔ Poster — окремо, через компоненти варіанта (розд. 2–4). Щоб один і той самий об’єкт мав сенс у всіх трьох місцях, **ID в компонентах на сайті** мають збігатися з тим, що потрапляє в **`CT_1008`** або в мапу цін (той самий `product_id` / `modificator_id` / `dish_modification_id`).

## 6. Узгодження даних адмінки сайту з Poster

Щоб рекалькуляція і фільтри (квіткові компоненти тощо) працювали передбачувано:

- Для **компонента без модифікатора і техкарти** у **`external_id`** має бути **`product_id`** Poster.
- Для **лінії з модифікатором** — заповнити **`modificator_id`** (і за потреби **`external_id`** = батьківський `product_id`, як у `poster.service` при підготовці списку).
- Для **лінії техкарти** — заповнити **`dish_modification_id`** і не плутати з `modificator_id`.

KeyCRM при цьому живе **паралельно**: той самий `modificator_id` / `dish_modification_id` / `product_id` має потрапляти в **`CT_1008`**, щоб синк знаходив пару «Poster ↔ KeyCRM».

## 7. Де це в коді (орієнтири)

- Рекалькуляція та побудова мапи цін Poster: `src/modules/product/product.service.ts` (`recalculatePrices`).
- Розбір меню для UI/списків компонентів: `src/modules/poster/poster.service.ts`.
- Синхронізація з KeyCRM: `src/lib/cron/sync-poster-keycrm.ts` (`CUSTOM_FIELD_UUID`, `INGREDIENT_ID_CUSTOM_FIELD_UUID`).

Документ відображає стан логіки на момент написання; при зміні полів Poster API або UUID кастомних полів KeyCRM потрібно оновити і цей файл, і відповідні константи в коді.
